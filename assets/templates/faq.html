Frequently asked questions

RIPACrypt isn't S3, or pastebin or dropbox so you've probably got a few questions...

HAVE QUESTIONS?

Why have storage that self destructs?

Destroying the crypt provides reasonable doubt as to whether you still possess the key required in a RIPA s.49 notice. It's also a useful failsafe for other purposes.

How do I get more storage space?

In order to store larger sized crypts (everyone gets 2Kb free) you will soon be able to transfer Bitcoin to the wallet address you were sent when you created your account.

Please note: We are not assigning a £ value to the storage we are simply saying that for every byte of storage you want to use you must transfer 1μBTC to us.

THIS FUNCTIONALITY IS NOT IN PLACE

Why is it so "expensive"?

Several reasons, the first is to discourage people from storing large amounts of data (this is supposed to be a password vault after all)

Secondly it is a one time cost that never expires. If for example you transferred 2048μBTC to the wallet ID you would have an extra 2Kb added to your storage quota. If you then stored 1964 bytes in a crypt it would stay live for as long as you continued to check-in. If the crypt is destroyed your storage quota is restored.

Finally the (btc)bits-for-bytes element is an easy yet anonymous anti-abuse strategy.

Are Bitcoin Deposits Refundable?

Under normal circumstances; No. You are not 'buying' anything. You are simply transfering some bytes to our control, be that data in your crypt or transactions on the blockchain.

What are the minimum / maximum check-ins / durations?

As of writing the minimum elapsed duration (check-ins x duration) is 3 hours (3x 1 hour), the maximum is a month (30x 1 day).

What is a crypt?

A crypt is the logical reference to the encrypted bytes you've stored along the associated meta-data. It is commonly described as a JSON object e.g.

{
 "status_code":201,
 "success":true,
 "status_message":"Crypt successfully created!",
 "version":1,
 "crypt":
 {
  "crypt_id": "29d48f65cf5899ba6119db5fc37648e5",
  "ciphertext":"W2+6uaow2WwHSitrOw1KmY9yqUykv41cHmdPtMF9pwP4Y9Q/eqOeqOv86+aFFoELgs4=",
  "crypt_timestamp":1464825599,
  "crypt_description":"Description of your crypt (preferably tying it the computer/secret you are protecting",
  "is_crypt_destroyed":false,
  "last_checkin":1464825599,
  "check_in_duration":86400,
  "miss_count":3
 }
}

What happens when too many check-ins are missed?

A watchdog process runs every few minutes which checks for any active crypts and then measures the last touch time.

If a crypt has missed the requisite number of check-ins the crypt is loaded into memory, the secret element is zeroed out, the crypt is set to read-only, the isDestroyed boolean is set to true and a destroyed time is set. The record is then saved back to the database.

RIPACrypt Policies

Law Enforcement Queries

We do not keep logs, we do not request or retain any identification data and we cannot decrypt the contents of a crypt.

Loss of Private Keys

If a GPG private key is list there is nothing we can do. Everything from authentication through to the secrets you store are protected by that key and our service never sees it.

DMCA et al Takedowns

Encrypted data stored within RIPACrypt is unlikely to be removed. This is due to the fact that only the owner can prove what it contains.

Anonymity

We do not store logs
We do not require any identification to set up or use our service.
We do not surrender any information without a court order
We do have a hidden service.

Data/Crypt Recovery

If a crypt is destroyed as scheduled there is no way to recover it. Data destruction is the purpose of this service. You should maintain (deniable) backups.
